# Outputs and Review

Each run writes a self-contained output directory. The files are designed so you can inspect OCR quality, rerun downstream stages, compare configuration choices, and reproduce the run later. Review the outputs before trusting a rendered overlay video.

## Output Directory

### Anatomy of an output directory

| File or directory | Description |
|---|---|
| `review/` | Sampled JPEG frames and `contact_sheet.jpg` for visual calibration review. The JPEGs include field bounding boxes. |
| `telemetry_raw.csv` | Raw OCR observations after parsing, including parse status and per-field details. |
| `telemetry_clean.csv` | Rebuilt telemetry table after filtering, stage logic, and anchor-point injection. This is the main retained dataset. |
| `telemetry_rejected.csv` | Points rejected by outlier processing, when available. |
| `trajectory.csv` | Dense trajectory reconstruction and augmented values, when trajectory is enabled. |
| `run_metadata.json` | Backend choice, sample settings, worker count, input video, output directory, and run context. |
| `config_resolved.yaml` | Snapshot of the profile used for the run. |
| `plots/` | PDF plots for retained telemetry, rejected points, trajectory, coverage, and stage summaries. |
| `telemetry_overlay.mp4` | Source video copy with synchronized telemetry plot, when overlay rendering is enabled. |
| `telemetry_overlay_preview.gif` | Lightweight overlay preview, when generated by the overlay path. |

### Review frames

Open `review/contact_sheet.jpg` first when diagnosing [calibration](calibration.md). It shows sampled frames with field boxes drawn over the source image.

Open individual JPEGs in `review/` when a field appears close to another overlay label or when the sampled frames show overlay movement. The boxes provide a visual check that OCR crops cover the intended digits and units.

Note: Review files are written to disk, but they are not listed among the finished output links in the run console. They can be inspected from the output directory when needed.

## Telemetry Tables

### Review raw telemetry

Open `telemetry_raw.csv` when you need to diagnose OCR behavior. It preserves raw text, units, parse status, and field-level observations used to build the clean table.

Use raw telemetry to answer questions such as:

- Did OCR see the digits at all?
- Was the unit label detected correctly?
- Did field fallback produce a better candidate than strip OCR?
- Did a hardcoded anchor point get injected?

### Review clean telemetry

Open `telemetry_clean.csv` for retained values by mission elapsed time. This table feeds plotting, trajectory reconstruction, and overlay rendering.

Core columns are:

| Column | Description |
|---|---|
| `mission_elapsed_time_s` | Time axis in seconds. |
| `stage1_velocity_mps` | Stage 1 velocity in meters per second, when available. |
| `stage1_altitude_m` | Stage 1 altitude in meters, when available. |
| `stage2_velocity_mps` | Stage 2 velocity in meters per second, when available. |
| `stage2_altitude_m` | Stage 2 altitude in meters, when available. |

Additional columns may be added by trajectory reconstruction.

### Review rejected points

Open `telemetry_rejected.csv` when a plot shows gaps or unexpected smoothing. Rejected points are preserved separately so you can decide whether the profile, parsing rules, or outlier thresholds need adjustment.

Use `webcalyzer reject-outliers` when you want to change rejection thresholds without redoing OCR. The command is documented in [CLI reference](cli-reference.md#rebuild-downstream-outputs).

## Trajectory and Media

### Review trajectory

Open `trajectory.csv` when trajectory reconstruction is enabled. It contains dense time steps, reconstructed velocity and altitude, acceleration, total distance, and downrange outputs. See [trajectory reconstruction](trajectory-reconstruction.md) for the assumptions behind these columns.

The dense trajectory treats mission elapsed time $t$ as the independent variable. Velocity $v(t)$ and altitude $h(t)$ are interpolated from clean telemetry, then distance is integrated over the grid.

For a trapezoid-style interval, the path increment is approximately:

$$
\Delta s_i \approx \frac{v_i+v_{i+1}}{2}\Delta t
$$

The downrange increment removes altitude change from that path increment:

$$
\Delta r_i = \sqrt{\max\left(0, \Delta s_i^2-\Delta h_i^2\right)}
$$

If launch-site latitude $\phi$, longitude $\lambda$, and azimuth $\alpha_{\mathrm{az}}$ are configured, downrange positions are projected with a WGS84 direct geodesic. Otherwise, webcalyzer reports a flat downrange distance.

### Review plots

Start with the summary PDFs under `plots/`. They show retained telemetry, rejected values when present, stage coverage, trajectory distance, and acceleration.

Rule of thumb: review plots before trusting the overlay video. The overlay is a presentation output, while the plots are better for inspecting noisy points and gaps.

### Review the overlay video

Open the configured overlay output filename, usually `telemetry_overlay.mp4`. The plotted values reveal over time, synchronized to the source video.

If the overlay is missing:

- confirm **Render overlay video** is enabled
- check whether the job logs show an ffmpeg or OpenCV failure
- try **Overlay engine** `opencv` if ffmpeg is not installed
- try **Overlay encoder** `libx264` if hardware encoding fails

### Reuse a resolved profile

Use `config_resolved.yaml` to reproduce a run. It contains the profile after web UI validation and YAML conversion.

For instance:

```bash
webcalyzer run --video /path/to/video.mp4 --config /path/to/output/config_resolved.yaml --output /path/to/new-output
```

Note: Run overrides such as OCR backend and overlay encoder are recorded in `run_metadata.json`, not in `config_resolved.yaml`, unless they were saved into the profile itself.
